## Summary

Статический аудит **ссылок и идентификаторов** в репозитории DocOps Agent после внешней проверки: markdown-ссылки в корневых файлах, поля метаданных (`Parent`, `Replaces`, при отсутствии — явные аналоги), смешение legacy и канонического формата метаданных, псевдо-ID и несоответствия **семейства имени файла / пути** и заявленного **doc_type**. Итог: **fail для машинной валидации без legacy-режима**; для человеческого чтения репозиторий согласован с оговорками, зафиксированными ниже. Изменения существующих документов в рамках этого шага **не выполнялись** (create-only через новые артефакты).

## Scope

- Каталог `docs/` (все `*.md`).
- Корневые markdown-файлы: `README.md`, `WORKFLOW.md`.
- Учтены шаблоны `templates/**/*.template.md` только как эталон канона (не как предмет «нарушений»).
- Не проверялись внешние URL и доступность сетевых ресурсов.

## Validation method

- Обход дерева `docs/` и корня; выборочное чтение файлов с разными паттернами заголовков и метаданных.
- Поиск по репозиторию: markdown-ссылки `\](...)`, поля `Parent`, `Replaces`, `Dependencies`, наличие блока `## Metadata`, строки `ID:` / `Doc type:` / `Type:`.
- Сверка заявленных в метаданных **ID** и **doc_type** с именем файла (`DOA-<FAMILY>-<NNN>`) и каталогом `docs/<doc_type>/`.
- Проверка разрешимости ссылок на ID документов: наличие файла с тем же логическим ID в метаданных или однозначный legacy-случай (описан в Findings).

## Findings

1. **Legacy `decision_log` без канонического блока метаданных.** Из **28** файлов в `docs/decision_log/` только **DOA-DEC-025** и **DOA-DEC-027** содержат блок `## Metadata` с полем `- ID:`. Остальные записи (например **DOA-DEC-004**, **DOA-DEC-028**, **DOA-DEC-030**) идентифицируются по заголовку вида `# Decision: DOA-DEC-…` или `# DOA-DEC-…` и **не дублируют ID в машиночитаемом поле** в стиле шаблона `templates/decision_log/decision_log.template.md`. Для автоматической валидации «ID = поле metadata» такие файлы **не являются источником истины** в канонической модели.

2. **Смешение канонического и альтернативного формата метаданных.** В **DOA-DEC-027** используется поле **`Type:`** вместо **`Doc type:`**, статус **`approved`** (вне перечня `draft` / `review` / `accepted` / `obsolete` из **DOA-ARCH-007**), в блоке метаданных **отсутствует `Parent`**, тогда как шаблон и архитектура требуют явного `Parent` (включая `null`). Это **несовместимо** с единым парсером без нормализации.

3. **`Parent` / `Replaces`, указывающие на неразрешимые или нестандартные ID.**
   - **DOA-ARCH-003**: `Parent: DOA-DOC-IDEA` — в репозитории **нет** документа с ID `DOA-DOC-IDEA`; ближайший канонический аналог по смыслу — **DOA-IDEA-001** (`idea_to_mvp`). Префикс **`DOC`** в семействе имени **не соответствует** соглашению `DOA-IDEA-*` / `DOA-ARCH-*`.
   - **DOA-OP-001** (`DOA-OP-001_v2_mvp_implementation_bootstrap.md`): `Replaces: DOA-OP-001 (draft)` — значение **не является** чистым ID (суффикс `(draft)`); машинное сопоставление с файлом/записью `DOA-OP-001` **неоднозначно**.

4. **Псевдо-ID и «строковые» ссылки в тексте (не поля metadata).** В **DOA-DEC-027** в теле перечислены исторические имена файлов (**`DOA-DEC-004_TZ_structure_fixed.md`**, **`DOA-IMP-008_project_snapshot_2026-04-08.md`**, **`DOA-OP-TZ-001_full_TZ_plan.md`**), которые **не совпадают** с текущими именами артефактов в `docs/` — для автоматической трассировки это **неразрешимые** указатели без таблицы соответствий.

5. **Несоответствие семейства имени / каталога и `doc_type`.**
   - **`docs/implementation_snapshot/DOA-OP-017.md`**: имя и заголовок относятся к **operational_plan** (**OP-017**), файл лежит в **`implementation_snapshot`**; метаданные в стиле TZ (**`## Project`**, **`## Parent`**) **без** полного канонического набора (**Doc type**, **ID**, **Date** в одном блоке). Уже отмечено в **DOA-AUD-001** / **DOA-AUD-002**; подтверждено при данном проходе.

6. **`operational_plan` без блока `## Metadata`.** Файлы **DOA-OP-011 … DOA-OP-016** содержат только краткий заголовок `# DOA-OP-…` и тело; **нет** полей Project / Doc type / ID / Status / Date / Parent. Путь `docs/operational_plan/` задаёт семейство **OP**, но для машины **doc_type и ID не извлекаются** из канонического блока.

7. **Legacy `architecture_draft` с ID только в H1.** **DOA-ARCH-001**, **DOA-ARCH-002** (и аналогичный стиль импорта TZ) используют строку вида `# architecture_draft — DOA-ARCH-00N` **без** блока `## Metadata`; канонический **`- ID:`** в начале файла **отсутствует**.

8. **Markdown-ссылки в корне.** В **`README.md`** есть относительные ссылки на **`WORKFLOW.md`** — цели существуют в корне репозитория. В **`WORKFLOW.md`** используются **внутристраничные** якоря (`#planning-rules`, `#decision-rules`) — проверка по тексту файла: якоря соответствуют заголовкам. В каталоге **`docs/`** markdown-ссылок вида `[text](path)` **не обнаружено** (трассировка между документами в основном через явные **ID** в тексте, не через относительные пути).

9. **Поле `Dependencies`.** Явных полей **`Dependencies:`** в проверенных документах `docs/` **не найдено**; риск для будущего контракта — отсутствие единого имени и правил для зависимостей при появлении таких полей.

10. **Двойная идентичность в заголовке vs метаданных (не ошибка, но шум для парсера).** Например **DOA-IMP-012**: H1 содержит **DOA-IMP-012**, блок метаданных дублирует **ID** — при парсинге необходимо правило приоритета: **metadata побеждает**.

## Severity

| Уровень | Наблюдение |
|--------|------------|
| **High** | Большинство `decision_log` и часть `operational_plan` / `architecture_draft` без канонического блока `## Metadata` с полем **ID** — автоматическая проверка ссылок по метаданным **невозможна** без legacy-режима. |
| **High** | Неразрешимые или нестандартные ссылки: **DOA-DOC-IDEA**, **Replaces: DOA-OP-001 (draft)**, псевдо-имена файлов в **DOA-DEC-027**. |
| **Medium** | **DOA-DEC-027**: поле **Type** vs **Doc type**, нет **Parent**, статус вне enum — ломает единый парсер. |
| **Medium** | **DOA-OP-017**: несоответствие каталога и типа документа + неполные метаданные. |
| **Low** | Отсутствие markdown-ссылек в `docs/` — низкий риск битых относительных путей между документами; основной риск — **семантические** ID, не файловые пути. |

## Risks

- Любой валидатор, который читает только **заголовок** или только **имя файла**, расходится с документами, где источник истины заявлен как **metadata** в новых файлах.
- Полуавтоматические отчёты (граф Parent / Replaces) дадут **ложные обрывы** или **дубликаты**, если не ввести явный **legacy mode** и реестр соответствий.
- Расширение полей (**Dependencies**) без контракта приведёт к новым несовместимым вариантам написания.

## Recommended actions

- Принять **decision_log** с каноническим контрактом ссылок и метаданных для машинной валидации (**см. следующий документ DOA-DEC-031**).
- Спроектировать валидатор: **основной режим** — парсинг блока `## Metadata`; **legacy mode** — эвристики (заголовок `# Decision: …`, H1 `architecture_draft — DOA-ARCH-…`, имя файла `DOA-<FAMILY>-<NNN>.md`) с явной маркировкой находок как **legacy**, без перезаписи старых файлов.
- В следующих **create-only** циклах: новые документы **строго** по шаблону; для исправления трассировки **DOA-DOC-IDEA** и **Replaces** — новые архитектурные/операционные записи, указывающие на канонические ID, **без** правки **DOA-ARCH-003** / **DOA-OP-001** в этом шаге.
- Унифицировать документацию поля зависимостей (**Dependencies**) при первой реальной потребности — отдельным решением и шаблоном.
