## Context

В рамках внешнего архитектурного аудита (`DOA-AUD-008`) использовались несколько систем (Qwen, Perplexity, OpenClaw-CODEX).

Qwen и Perplexity смогли выполнить аудит при передаче ссылок на репозиторий.

OpenClaw-CODEX не смог выполнить строгий аудит, несмотря на доступ к репозиторию, по причине отсутствия явного текстового содержимого документов во входных данных.

## Problem

Передача только ссылки на репозиторий или указание путей к файлам не гарантирует, что внешний аудитор:

- сможет получить содержимое документов,
- сможет использовать его как верифицируемый источник,
- сможет выполнить section-level доказательный аудит.

Это нарушает требование строгого аудита:

- каждое утверждение должно ссылаться на конкретный документ и его раздел,
- входные данные должны быть явно доступны в пределах текущего прогона.

## Decision

Внешний аудит **MUST** получать артефакты в виде **явного входа** (explicit artifact input).

Допустимые форматы передачи:

1. Полный текст документов, переданный в prompt
2. Прямые raw-ссылки на файлы (например, `raw.githubusercontent.com`)
3. Архив (zip) с документами, предоставленный аудитору

Передача только:

- ссылки на репозиторий
- путей к файлам внутри репозитория

**НЕ** считается достаточной для строгого аудита.

## Constraints

- Решение применяется **только** к внешнему аудиту.
- **Не** изменяет архитектурные документы (`DOA-ARCH-018` / `019` / `020`).
- **Не** влияет на validator / gate / Report Layer.

## Rejected approaches

1. **«Аудитор сам должен уметь читать репозиторий»** — отклонено: поведение моделей различается, не гарантируется.

2. **«Достаточно указать пути к файлам»** — отклонено: отсутствует гарантия доступа к содержимому.

3. **«Можно допускать предположения аудитора»** — отклонено: противоречит строгому аудиту (no assumptions).

## Consequences

**Позитивные:**

- гарантируется воспроизводимость аудита;
- обеспечивается section-level доказательность;
- устраняется зависимость от поведения конкретной модели.

**Негативные:**

- увеличивается объём входных данных;
- требуется подготовка audit input (snapshot / bundle).

## Follow-ups

- **MAY** быть создан отдельный документ для стандартизации audit input snapshot (например, `audit_input_bundle`).
- **MAY** быть добавлен шаблон для передачи документов во внешний аудит.
