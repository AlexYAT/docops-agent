# Decision: DOA-DEC-030
Date: 2026-04-08
Status: accepted

## Title
Добавить doc_type `other` и doc_type `rules` в DocOps Agent MVP

## Context
В текущей архитектуре DocOps Agent набор doc_types покрывает core и planning документы, но не покрывает:
- вспомогательные проектные материалы, которые полезно хранить в репозитории и на которые нужно ссылаться из других документов
- операционные правила работы, которые уже фактически существуют, но не оформлены как отдельный согласованный doc_type

Сейчас `README.md`, `WORKFLOW.md`, `PROJECT_CONTEXT.md` и файл commit rules существуют как полезные артефакты, но не все они естественно укладываются в текущий набор doc_types. Это создаёт “серую зону” между канонической архитектурой и практикой проекта.

## Options considered
1. Оставить такие файлы вне модели doc_types
2. Относить их к существующим `decision_log` / `architecture_draft`
3. Ввести новые doc_types: `other` и `rules`

## Chosen solution
Ввести два новых doc_types:
- `other` — для вспомогательных артефактов, на которые можно ссылаться из других документов, но которые не являются core/planning типами
- `rules` — для проектных правил работы, commit discipline, process rules и аналогичных нормативных документов

## Why
- уменьшает расхождение между фактической практикой и архитектурой
- позволяет хранить служебные и процессные документы без “ломания” существующих doc_types
- улучшает traceability: на такие документы можно ссылаться явно
- делает архитектуру более честной и пригодной для дальнейшего развития

## Trade-offs
- архитектура становится немного шире
- потребуется обновить каталог `docs/`, `templates/`, naming/metadata и workflow
- возможно появление соблазна складывать в `other` слишком многое без дисциплины

## Guardrails
- `other` использовать только для артефактов, которые не подходят под существующие doc_types
- `rules` использовать только для нормативных и процессных документов
- `other` не должен превращаться в “свалку”
- все новые документы по-прежнему обязаны иметь metadata, naming и parent/null

## Follow-ups
- выпустить новую версию architecture_draft с добавлением `other` и `rules`
- создать каталоги `docs/other`, `docs/rules`, `templates/other`, `templates/rules`
- определить naming mapping для `OTHER` и `RULES`
- обновить WORKFLOW.md и README при необходимости
