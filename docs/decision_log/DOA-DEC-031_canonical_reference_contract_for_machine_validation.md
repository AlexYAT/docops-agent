## Title

Канонический контракт ссылок и метаданных для машинной валидации (reference contract)

## Context

По результатам внешнего аудита ссылок и внутреннего обхода репозитория (**DOA-AUD-003**) выявлено: исторические документы используют ID в заголовке или в имени файла без единого блока `## Metadata`; часть полей (`Type` vs `Doc type`, нестандартные значения Parent/Replaces) не позволяет однозначно строить граф ссылок без специальных правил. Необходимо зафиксировать **источник истины** и правила для **новых** документов и для **валидатора**, не нарушая политику create-only для уже существующих файлов.

## Options considered

1. **Объявить заголовок или имя файла единственным SoT** — просто для скриптов, но противоречит принятой модели **DOA-ARCH-007** и шаблонам T03.
2. **Массово переписать legacy-документы** — нарушает create-only и запрет ретро-правок в текущем цикле.
3. **Зафиксировать metadata как SoT для новых документов + явный legacy mode для валидатора** — согласуется с архитектурой и аудитом.

## Chosen solution

Принять следующий **канонический контракт** для машинной валидации и ручной дисциплины:

1. **Блок `## Metadata` в документе — машиночитаемый источник истины** для полей, которые в нём присутствуют: при разночтении с заголовком H1 или с пояснениями в теле **приоритет у метаданных** (для документов, где этот блок есть и содержит поле `ID`).
2. **Все новые документы** под `docs/<doc_type>/` **обязаны** включать блок `## Metadata` по соответствующему шаблону из `templates/`, с обязательным полем **`- ID:`** в формате проекта (**`DOA-<FAMILY>-<NNN>`**), согласованным с **Doc type** и размещением в каталоге.
3. **Legacy-документы** (созданные до этого контракта или без полного блока метаданных) **допускаются** в репозитории без обязательной миграции; они должны учитываться валидатором как **legacy** (эвристики: заголовок `# Decision: DOA-DEC-…`, H1 `… — DOA-ARCH-…`, согласование с именем файла). Legacy-режим **временный**: его цель — постепенное вытеснение за счёт **новых** записей, а не перезапись истории.
4. **Поля трассировки `Parent`, `Replaces` и (при введении) `Dependencies`** должны содержать **разрешимые идентификаторы** существующих документов репозитория (по полю `ID` в метаданных целевого документа или, в legacy mode, по согласованной эвристике), либо явное **`null`** где это предусмотрено шаблоном. Недопустимы произвольные строки, не совпадающие с ID (**пример анти-паттерна**: псевдо-ID вроде `DOA-DOC-IDEA` без соответствующего документа; **Replaces** с комментариями в скобках без отдельного канонического ID).
5. Валидатор **может** поддерживать **legacy mode** до отдельного решения о его отключении; в отчётах legacy-находки помечаются отдельно от нарушений канона в **новых** файлах.

## Why

- Снимает неоднозначность для автоматизации и внешних аудитов без отмены create-only.
- Согласуется с **DOA-ARCH-007** (metadata model) и с выводами **DOA-AUD-003**.
- Разделяет «исторически допустимо» и «обязательно для нового».

## Trade-offs

- До выключения legacy mode валидатор сложнее и требует поддержки эвристик.
- Полное совпадение графа ссылок с бизнес-смыслом достигается постепенно через новые документы, а не одним массовым исправлением.

## Follow-ups

- Реализовать или обновить скрипт валидации (канон + legacy) и зафиксировать его поведение в **implementation_snapshot** или **operational_plan** отдельным create-only шагом.
- При введении поля **Dependencies** — обновить шаблоны и данное решение ссылкой **create-only** новым **decision_log**, если потребуется уточнение семантики.
