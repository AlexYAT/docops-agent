## Title

Стандарт запроса внешнего аудита архитектуры для сопоставимости результатов аудиторов

## Context

Внешний аудит архитектуры DocOps Agent выполняется **несколькими** аудиторами (например различные LLM / системы). Документ `DOA-AUD-008` зафиксировал консолидацию без приложенных первичных отчётов аудиторов, то есть **входные данные аудиторов отсутствовали** при фиксации.

## Problem

- Ответы разных аудиторов **несопоставимы**, если запрос сформулирован по-разному или без явного scope.
- **Нет единого стандарта** запроса — невозможно агрегировать выводы по единым критериям.
- Риск **субъективных интерпретаций** и выводов без привязки к тексту документов репозитория.

## Options considered

1. **Свободный запрос** — каждый аудитор получает общее описание задачи без жёсткой структуры вывода. Отклонён: снижает сопоставимость.
2. **Стандартизированный запрос** — единый шаблон роли, scope, ограничений, обязательных проверок и формата ответа. **Принят.**

## Chosen solution

Для каждого внешнего архитектурного аудита **MUST** использоваться **единый** шаблон запроса (**audit request template**), описанный в разделе «Audit request template». Запросы, не содержащие обязательных блоков, **MUST NOT** считаться допустимым входом для консолидированного `audit_check`, ожидающего сопоставимые ответы.

## Audit request template

Каждый запрос к внешнему аудитору **MUST** включать следующие блоки **в указанном порядке смысла** (формат передачи — вне объёма настоящего решения):

### Role

- Явное указание роли: **аудитор архитектуры** (только проверка заявленных артефактов; не проектировщик, не реализатор).

### Scope

- Перечень **идентификаторов и/или путей** документов репозитория, подлежащих проверке (например `DOA-ARCH-018`, путь к файлу).
- Запрет расширения scope без явного изменения запроса.

### Constraints

- Что **запрещено** аудитору, например:
  - выдумывать факты о репозитории;
  - выходить за перечень Scope;
  - предлагать реализацию, код, изменения файлов;
  - делать выводы без ссылки на конкретный документ и раздел/цитату по возможности.

### Required checks

Аудитор **MUST** явно ответить по каждому пункту (даже если «не применимо» — с обоснованием):

1. **Policy Contract consistency** — согласованность и полнота норм Policy Contract в пределах Scope.
2. **Validator / Gate alignment** — соответствие описаний validator и gate единому контракту (по тексту документов в Scope).
3. **Report Layer anti-SoT** — соблюдение инвариантов: отчёт не SoT, не источник policy, ограничения по `Parent`/`Replaces`/фактам (по тексту документов в Scope).
4. **DoD verifiability** — для затрагиваемых решений о DoD: критерии объективно проверяемы, субъективные критерии не выдаются за обязательные (по тексту документов в Scope).

### Output format

Ответ аудитора **MUST** содержать разделы:

1. **Findings** — наблюдения; каждый пункт с привязкой к документу (`DOA-*` и при необходимости путь).
2. **Violations** — только явные нарушения инвариантов **по тексту** документов; если нет — явно «нет».
3. **Verdict** — одно значение: `PASS` / `CONDITIONAL` / `FAIL` (словарь фиксирован).
4. **Justification with references** — обоснование вердикта со ссылками на документы и фрагменты утверждений.

## Constraints for auditors

- **MUST NOT** придумывать факты о содержимом репозитория вне предоставленного Scope.
- **MUST NOT** выходить за Scope.
- **MUST NOT** предлагать реализацию, патчи, задачи в трекере как обязательный результат аудита.
- **MUST** сопровождать каждое содержательное утверждение ссылкой на документ(ы) из Scope.

## Expected output format

- **Строгая структура** разделов, совпадающая с подразделом «Output format» шаблона запроса.
- **Без свободного нарратива** вне перечисленных разделов (вводный абзац ≤ 2 предложений допускается только для идентификации прогона).
- **Обязательные ссылки** на документы (`DOA-*` и/или путь) в каждом пункте Findings и в Justification.

## Rejected approaches

- **Свободный аудит** без фиксированного Output format — отклонён.
- Итог в форме **«общего мнения»** без Verdict и без ссылок — отклонён.

## Consequences

- Повышается **сопоставимость** ответов разных аудиторов по одним и тем же проверкам.
- Упрощается **агрегация** в последующем `audit_check` (табличное слияние по Required checks и Verdict).

## Follow-ups

- Сформировать новый `audit_check` с консолидацией **после** получения ответов аудиторов, сформированных по настоящему шаблону (create-only).
- При необходимости уточнить шаблон отдельным `decision_log` или `architecture_draft`, если Scope аудита расширится (create-only).
