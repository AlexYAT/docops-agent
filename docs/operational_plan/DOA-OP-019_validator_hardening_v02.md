## Goal

Усилить **link/ID validator** до **v0.2** (hardening): закрыть пробелы **v0.1.0**, зафиксированные в **DOA-IMP-013** — детекция несоответствия папки и семейства ID для **legacy**-файлов, устойчивый реестр ID, проверка markdown-ссылок в `docs/`, нормализация категорий и политики severity, уточнение отчёта; затем **второй прогон** и новый **implementation_snapshot**. Существующие документы в `docs/` **не переписываются** (create-only дисциплина сохраняется).

## Scope

### Includes

- доработка `scripts/doa_link_id_validator.py` (версия **0.2.x**, уточнённый schema при необходимости);
- новый machine-readable отчёт второго прогона (отдельный файл под `reports/`);
- новый **implementation_snapshot** по итогам T06.

### Excludes

- правка legacy markdown в `docs/` и `templates/` для «зелёного» отчёта;
- массовая миграция метаданных;
- проверка внешних HTTP(S) URL и сетевой доступности;
- обязательная интеграция в CI (может следовать отдельным планом).

## Inputs

- **DOA-OP-018** — исходный план внедрения v0.1 и состав задач T01–T06.
- **DOA-IMP-013** — результаты первого прогона, метрики, наблюдения (в т.ч. **DOA-OP-017** и реестр ID).
- **scripts/doa_link_id_validator.py** v0.1.0 — текущая реализация и отчёт `doa-validator-report/1`.
- **DOA-DEC-031** — canonical vs legacy, приоритет metadata.
- **reports/doa_link_id_validator_run_001.json** — baseline для сравнения второго прогона.

## Tasks

### T01 — Legacy folder/family mismatch detection

**Описание:** Для файлов **без** канонического `## Metadata` + `ID` извлекать кандидатный ID из имени файла и/или заголовка; если семейство (**OP** / **IMP** / **DEC** / …) **не соответствует** каталогу `docs/<doc_type>/`, эмитить отдельную **legacy**-категорию (например `legacy_folder_family_mismatch`), покрывая кейс **`DOA-OP-017.md`** в `implementation_snapshot/`.

**Done when:** Отчёт v0.2 стабильно помечает согласованные несоответствия без ложных срабатываний на файлы без префикса `DOA-*` в имени.

---

### T02 — Duplicate / ambiguous ID registry handling

**Описание:** Уточнить построение реестра: явно детектировать **дубликаты** (один ID → несколько файлов), **конфликт источников** (metadata ID ≠ stem/header), **неоднозначный** разбор; выводить в отчёте структурированные записи с перечнем путей и типом конфликта.

**Done when:** Категории вроде `duplicate_id_registry` / `ambiguous_id_source` документированы в коде или README скрипта и появляются в JSON при синтетическом или реальном конфликте.

---

### T03 — Deeper markdown link validation in docs/

**Описание:** Расширить проверку `[text](target)` на все файлы под `docs/**/*.md`: относительные пути от файла, якор-only ссылки, игнор внешних URL; опционально — валидация, что целевой `.md` существует (без обязательной проверки якоря в целевом файле на первом шаге).

**Done when:** Для выборки ссылок в `docs/` отчёт содержит `broken_markdown_link` или эквивалент с путём источника и целью; ложные срабатывания на `http(s)://` отсутствуют.

---

### T04 — Category normalization and severity policy

**Описание:** Ввести стабильный **каталог кодов** категорий (canonical vs legacy); для каждой записи задать **severity** (`error` / `warn` / `info`) и **mode** (`canonical` / `legacy` / `root`); задокументировать политику: что считается блокирующим для будущего CI vs информационным.

**Done when:** В JSON каждая находка содержит нормализованный `category` из реестра и `severity`; в шапке отчёта — `schema` версия ≥ `doa-validator-report/2` **или** явное поле `policy_version`.

---

### T05 — Report format refinement

**Описание:** Уточнить JSON: сводки по режимам, `top_categories` с разбивкой canonical/legacy, опционально краткий `summary` object для человека; сохранить обратную совместимость ключей `canonical_violations` / `legacy_findings` или описать миграцию в одном месте.

**Done when:** Второй отчёт записан как новый файл (например `reports/doa_link_id_validator_run_002.json`) и читается без ручной догадки о структуре.

---

### T06 — Second validator run + implementation_snapshot

**Описание:** Выполнить полный прогон v0.2 на том же охвате, что и v0.1 (`docs/**/*.md`, `README.md`, `WORKFLOW.md`); сохранить отчёт; создать **новый** `implementation_snapshot` с уникальным ID, **Parent:** **DOA-OP-019**, сравнением с **DOA-IMP-013** и списком новых категорий/метрик.

**Done when:** Существуют артефакт отчёта и новый файл **IMP** в `docs/implementation_snapshot/`, закрывающий T06.

## Success criteria

- Версия валидатора поднята до **0.2.x**; в отчёте указаны версия и schema/policy.
- **Legacy** folder/family mismatch детектируется для типичного misplaced кейса (**OP** в **implementation_snapshot**).
- Markdown-ссылки в `docs/` проверяются глубже, чем в v0.1 (где в `docs/` ссылок практически не было — механизм готов к росту ссылочности).
- Категории и **severity** согласованы и документированы; отчёт второго прогона сохранён в `reports/`.
- Создан новый **implementation_snapshot** после второго прогона.

## Risks

- Новые эвристики дают шум на нестандартных именах файлов — нужны тесты на граничные случаи и явные исключения только по решению (не silent).
- Ужесточение ссылок может выявить исторические битые пути — ожидаемо; исправление только через новые документы или отдельные планы.
- Расхождение v1 и v2 schema отчёта усложнит сравнение с `run_001` — минимизировать за счёт явного `summary` и счётчиков.

## Out of scope

- Редактирование существующих `docs/**/*.md` для устранения находок.
- Автоматический fail pipeline в CI на legacy (только по отдельному решению).
- Парсинг нескольких блоков `## Metadata` в одном файле как полноценная модель документа (допускается ограниченная эвристика или предупреждение — по усмотрению T01/T04, без обязательной полной поддержки).
