## Goal

Внедрить **link/ID validator** для репозитория DocOps Agent: автоматизируемая проверка метаданных и ссылочных полей с разделением **канонических нарушений** и **legacy-находок**, в соответствии с **DOA-DEC-031** и выводами **DOA-AUD-003**.

## Scope

### Includes

- инвентаризация поддерживаемых полей ссылок и идентификаторов;
- правила парсинга блока `## Metadata` для канонических документов;
- эвристики legacy mode для документов без полного metadata-блока;
- каталог категорий проверок и формат отчёта;
- первый прогон валидатора и фиксация результатов в **implementation_snapshot** (отдельным артефактом после выполнения плана).

### Excludes

- правка существующих документов для «исправления» находок валидатора;
- массовая миграция legacy в канонический формат;
- CI/CD-интеграция (может следовать отдельным планом);
- валидация внешних URL и сетевых ресурсов.

## Inputs

- **DOA-AUD-003** — перечень проблем и метод статической проверки.
- **DOA-DEC-031** — canonical reference contract (metadata SoT, legacy mode, resolvable Parent / Replaces / Dependencies).
- **DOA-ARCH-007** — модель обязательных и опциональных полей metadata.
- Шаблоны `templates/**/*.template.md` — эталон имён полей (`Doc type`, `ID`, `Parent`, `Replaces`, и т.д.).
- Текущее дерево `docs/` и корневые `README.md`, `WORKFLOW.md`.

## Tasks

### T01 — Inventory of supported reference fields

**Описание:** Зафиксировать список полей, которые валидатор читает для ссылок и идентичности (минимум: `ID`, `Doc type` / распознавание алиаса `Type`, `Parent`, `Replaces`; опционально заготовка под `Dependencies`).

**Done when:** Есть короткая таблица «поле → обязательность → допустимые значения / формат ID» в репозитории кода или в сопроводительной заметке к скрипту (как часть реализации T02–T05).

---

### T02 — Canonical metadata parsing rules

**Описание:** Реализовать разбор блока `## Metadata`: извлечение пар ключ–значение, нормализация регистра ключей, приоритет **metadata над H1** при наличии `- ID:`.

**Done when:** Парсер стабильно извлекает канонические поля из файлов вида **DOA-DEC-025**, **DOA-OP-002** без ручной подстройки под каждый файл.

---

### T03 — Legacy mode heuristics

**Описание:** Для файлов без полного `## Metadata`: извлечь кандидатный ID из `# Decision: DOA-DEC-…`, `# DOA-DEC-…`, H1 `… — DOA-ARCH-…`, имени файла `DOA-<FAMILY>-<NNN>.md`; помечать каждую находку как **legacy**, не смешивать с каноническим fail без явного флага режима.

**Done when:** Для выборки из **DOA-AUD-003** (decision_log без metadata, **DOA-ARCH-001/002**, **DOA-OP-011…016**) валидатор выдаёт предсказуемую legacy-классификацию.

---

### T04 — Validation categories

**Описание:** Ввести категории проверок, включая как минимум:

- отсутствие или несогласованность **ID** (нет поля, расхождение с именем файла / семейством);
- **неразрешённый Parent** (ID цели отсутствует в репозитории);
- **невалидный Replaces** (не чистый ID, ссылка на несуществующий документ, нарушение правила «тот же doc_type» при строгом режиме);
- **несоответствие папки и doc_type** (например `DOA-OP-*` под `docs/implementation_snapshot/`);
- несоответствие **Doc type** и каталога `docs/<doc_type>/`;
- альтернативные ключи metadata (**`Type:`** vs **`Doc type:`**), значения статуса вне enum — как **canonical warning** или **legacy**, по правилам из T02/T03.

**Done when:** Каждая категория имеет код/тип в отчёте и описание в одном месте (комментарий к скрипту или мини-спека рядом).

---

### T05 — Output / report format

**Описание:** Определить формат вывода (например machine-readable JSON + человекочитаемый summary): путь файла, режим **canonical / legacy**, категория, сообщение, ссылочное поле.

**Done when:** Один прогон воспроизводимо генерирует отчёт, пригодный для приложения к **implementation_snapshot**.

---

### T06 — First validator run snapshot

**Описание:** Выполнить полный прогон по `docs/` и корневым md; сохранить артефакт отчёта; оформить **implementation_snapshot** с кратким резюме прогона и ссылкой на файл отчёта (следующий артефакт после закрытия этого плана).

**Done when:** Создан новый **implementation_snapshot** с новым уникальным ID, описывающий первый прогон (см. **Next artifact**).

## Success criteria

- Валидатор в каноническом режиме проверяет документы с блоком `## Metadata` и полем **ID** по правилам **DOA-DEC-031**.
- В отчёте **явно различаются** записи: **canonical violation** (нарушение контракта для канонически распознанного документа) и **legacy finding** (наследие формата или эвристическое разрешение без переписывания файлов).
- Категории из **T04** покрывают кейсы, перечисленные в **DOA-AUD-003** (missing ID, unresolved Parent, invalid Replaces, folder/type mismatch и др.).
- Первый полный прогон задокументирован в **implementation_snapshot** (после выполнения **T06**).

## Risks

- Эвристики legacy дадут ложные совпадения ID или пропуски — нужна явная маркировка **legacy** и пересмотр правил без правок старых md.
- Расхождение «строгий канон vs мягкий legacy» может путать читателей отчёта — в отчёте обязательны раздельные секции или поля.
- Дубли ID между заголовком и metadata могут маскировать ошибки, если не применять правило приоритета metadata.

## Out of scope

- Изменение содержимого уже существующих документов в `docs/` и `templates/` для устранения находок.
- Введение новых doc_types или изменение **DOA-ARCH-007** без отдельного архитектурного цикла.
- Автоматический fail CI на legacy-находках (только по отдельному решению).

## Next artifact
