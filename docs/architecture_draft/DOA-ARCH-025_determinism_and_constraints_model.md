## 1. Summary

Фиксация **constraint-based** модели DocOps, обеспечивающей детерминизм и воспроизводимость системы: канонические пути, неизменность префикса, жёсткий lifecycle, семантика IMP и воспроизводимость prompt execution.

Документ принят в рамках цикла hardening после **DOA-IDEA-005** и относится к контексту boundary **DOA-FSN-002**.

---

## 2. Canonical Path Model

### Определение

Пути документов определяются **детерминированно** с приоритетом **уже существующей** структуры репозитория: размещение нового артефакта того же `doc_type` выводится из наблюдаемого канона (наличие файлов того же типа в одном корневом каталоге под `docs/`).

### Режимы

#### Derive Mode (основной)

- путь берётся из существующих документов того же `doc_type` (наблюдаемый единый корневой каталог);
- **новые каталоги создавать запрещено**.

#### Define Mode (если пути ещё нет)

- если документов данного типа нет, путь создаётся по шаблону:

`docs/<doc_type>/`

где `<doc_type>` — значение поля **Doc type** в metadata (slug каталога **строго совпадает** с `doc_type`).

Примеры типов и каталогов для DocOps Agent (в т.ч. целевые при появлении первого документа типа):

- `idea_to_mvp` → `docs/idea_to_mvp/`
- `architecture_draft` → `docs/architecture_draft/`
- `decision_log` → `docs/decision_log/`
- `operational_plan` → `docs/operational_plan/`
- `implementation_snapshot` → `docs/implementation_snapshot/`
- `validation_reports` → `docs/validation_reports/` (только после появления первого документа этого типа и явной политики в decision_log при необходимости)

### Правила

- создание нового каталога допустимо **только** в **Define Mode** и **только** как `docs/<doc_type>/` с именем, совпадающим с `doc_type`;
- **запрещены** альтернативные названия каталогов и «удобные» сокращения;
- после создания путь становится **каноническим** для данного типа.

### Инвариант

**Один тип документа (`doc_type`) → один каталог** под `docs/`.

## 3. Prefix Model

- используется **единый** prefix проекта: **DOA** (в ID вида `DOA-<KIND>-<NNN>`);
- **запрещено** создавать новые prefix;
- **запрещено** смешение prefix в рамках одной трассируемой цепочки артефактов.

---

## 4. Lifecycle Model

Канонический lifecycle:

`ANALYZE → IDEA → ARCH → ANALYZE → DEC → OP → IMP → VALIDATION`

### Правила

- **оба** этапа **ANALYZE** обязательны (перед **IDEA** и перед **DEC**);
- **запрещено** пропускать стадии и сокращать цепочку.

---

## 5. IMP Semantics

- **IMP** фиксирует **факт** реализации (что сделано, на каком основании, какие артефакты);
- **запрещено** создавать IMP без опоры на реализацию и проверяемые следы (код, отчёты валидации, и т.п. — по политике OP/DEC);
- статус, означающий завершённую реализацию, допустим **только** при наличии согласованной с документом фактической реализации.

---

## 6. Prompt Reproducibility

Основание: **DOA-DEC-059** (принцип воспроизводимости prompt).

- **prompt** = единица изменения системы в управляемом исполнении;
- prompt должен быть **самодостаточным** для повторного выполнения;
- результат должен быть **воспроизводим** тем же prompt;
- **ручные доработки** вне prompt execution **запрещены** (см. правила DOA-DEC-059).

---

## 7. Execution Model

Роли (логическая модель):

- **ChatGPT** — формирует prompt (спецификация единицы изменения);
- **Cursor** — создаёт и размещает документы/артефакты в канонических путях согласно prompt и политике;
- **Человек** — validation, принятие решений, исполнение шагов вне автоматизируемой части.

---

## 8. Constraints

- нельзя создавать **произвольные** каталоги вне режимов §2;
- нельзя менять **prefix** и схему ID вне decision_log;
- нельзя **нарушать lifecycle** (§4);
- нельзя создавать **IMP без реализации** (§5);
- нельзя **пропускать ANALYZE** (§4).

---

## 9. Consequences

- рост **предсказуемости** размещения и эволюции артефактов;
- **снижение гибкости** свободного поведения LLM вне контрактов;
- улучшение **auditability** и трассируемости (parent, типы, пути).
