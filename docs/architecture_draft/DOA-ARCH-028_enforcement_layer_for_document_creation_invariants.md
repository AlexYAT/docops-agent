## Summary

This document describes the minimal architecture for enforcing document creation invariants in DocOps Agent.

The enforcement layer is intended to detect violations of document creation rules without modifying existing documents.

---

## Context

DocOps Agent already defines document creation invariants:

- doc_type determines folder
- ID is generated by repository scan
- filename follows naming convention
- document creation follows a controlled workflow

However, these rules are currently enforced by discipline only.

DOA-IDEA-007 introduces the need for a minimal enforcement layer.

---

## Architectural Goal

Provide a minimal detection-only enforcement layer that can validate whether newly created documents follow the expected creation invariants.

The architecture must preserve:

- create-only semantics
- MVP simplicity
- no breaking changes
- no automatic document mutation

---

## Enforcement Targets

The enforcement layer should validate:

1. doc_type → folder consistency
2. ID format correctness
3. filename naming convention
4. metadata consistency
5. basic creation workflow compliance

---

## Proposed Architecture

### 1. Validator as primary entry point

The existing validator should remain the primary enforcement entry point.

Responsibilities:

- scan documents
- detect invariant violations
- report findings
- avoid document mutation

### 2. Document Gate as optional control layer

Document Gate may consume validator output later.

Responsibilities:

- aggregate validation results
- classify status
- support acceptance / warning / rejection logic

Document Gate should not become the first implementation target unless required by a later DEC.

### 3. Enforcement report

The enforcement layer should produce a structured report.

Report should include:

- checked document path
- detected doc_type
- expected folder
- actual folder
- ID validation result
- filename validation result
- metadata validation result
- severity

---

## Flow

1. A document is created through Cursor
2. Validator scans the repository
3. Enforcement checks are applied
4. Violations are reported
5. User decides next action through a new DocOps document if needed

---

## Non-Goals

- no automatic fixes
- no rewriting existing documents
- no pre-commit hooks at this stage
- no CI integration at this stage
- no external system integration
- no full policy engine

---

## Open Questions

- Should enforcement be added directly to the existing validator?
- Should enforcement become a separate validator mode?
- Should Document Gate consume enforcement findings immediately or later?
- What severity should be assigned to each invariant violation?
