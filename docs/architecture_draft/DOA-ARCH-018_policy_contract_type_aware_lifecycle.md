## Title

Policy Contract: Type-Aware Lifecycle, Validator, and Gate

## Purpose

Зафиксировать единый **Policy Contract** — нормативные правила жизненного цикла статусов по `doc_type`, трактовки validator и gate, а также нормализации severity. Документ является **архитектурным каноном**; детали реализации, кода и автоматизации в объём не входят.

## Scope

- Статусы и допустимые переходы **по каждому** охваченному `doc_type`.
- Обязанности validator и gate относительно одних и тех же определений политики.
- Правила нормализации severity до уровня, на котором gate **MUST NOT** вводить альтернативную шкалу.
- Границы: что входит в политику жизненного цикла и метаданных статуса; что выходит (в т.ч. Report Layer — только граница: производные отчёты **MUST NOT** расширять или подменять политику статусов).

## Non-goals

- План внедрения, задачи, сроки, владельцы.
- Код, скрипты, CLI, API, схемы БД, форматы файлов отчётов.
- UI и сценарии взаимодействия пользователя.
- Семантика содержимого документов (текст решений, планов и т.д.), кроме полей, явно входящих в контракт (например `Status`, `doc_type`).

## Problem statement

При неявном или глобальном перечне статусов validator и gate расходятся в трактовке; severity может интерпретироваться по-разному. Это приводит к ложным срабатываниям (например статус, допустимый для одного `doc_type`, отклоняется как «невалидный» глобально) и к непредсказуемым итогам gate. Политика **MUST** быть единой и общей для validator и gate.

## Policy contract role

- **Policy Contract** — единственный архитектурный источник норм для: множества статусов по `doc_type`, допустимых переходов, правил интерпретации validator, правил интерпретации gate и нормализации severity (в границах, описанных ниже).
- Validator **MUST** применять только правила, выводимые из контракта для разрешённого `doc_type`.
- Gate **MUST** применять только правила, выводимые из того же контракта и из **нормализованного** вывода validator; gate **MUST NOT** подменять собственной логикой определения допустимости статуса по `doc_type`.

## Type-aware status model

- Система **MUST NOT** определять и **MUST NOT** использовать один универсальный enum статусов для всех `doc_type` как нормативную модель жизненного цикла.
- Для каждого охваченного `doc_type` множество допустимых значений статуса **MUST** задаваться явно в контракте (ниже).
- Значение статуса **MUST** оцениваться только в паре **(doc_type, status)**; применение правил другого `doc_type` **MUST NOT** допускаться.
- Статус `planned` **MUST** считаться допустимым только для planning-`doc_type` перечисленных в контракте; для прочих `doc_type` значение `planned` **MUST NOT** считаться валидным в рамках данного контракта.

## Allowed transitions by doc_type

Норма: переход — смысловая допустимость смены состояния **для политики** (включая сопоставление с цепочками `Parent` / `Replaces` на уровне правил контракта, без алгоритмов реализации). Терминальные состояния не имеют исходящих переходов в рамках одного логического артефакта.

### Planning documents (`operational_plan`, `midterm_plan`, `longterm_plan`)

- Допустимые статусы: `draft`, `planned`, `active`, `superseded`.
- Допустимые переходы: `draft` → `planned`; `planned` → `active`; `active` → `superseded`; `planned` → `superseded`; `draft` → `superseded`.

### `implementation_snapshot`

- Допустимые статусы: `draft`, `recorded`, `accepted`.
- Допустимые переходы: `draft` → `recorded`; `recorded` → `accepted`.

### `decision_log`

- Допустимые статусы: `draft`, `accepted`, `rejected`, `superseded`.
- Допустимые переходы: `draft` → `accepted`; `draft` → `rejected`; `draft` → `superseded`; `accepted` → `superseded`.

### `architecture_draft`

- Допустимые статусы: `draft`, `accepted`, `superseded`.
- Допустимые переходы: `draft` → `accepted`; `draft` → `superseded`; `accepted` → `superseded`.

### Иные `doc_type`

- Для `doc_type`, не перечисленных выше, набор статусов и переходов в рамках **данного** контракта **MUST** считаться не заданным; validator и gate **MUST NOT** выводить для них нормы из глобального enum; расширение политики **MUST** выполняться отдельным архитектурным артефактом (create-only), с явным обновлением контракта.

## Validator contract

- Validator **MUST** определить `doc_type` документа (как минимум по размещению в `docs/<doc_type>/` и согласованно с метаданными, если метаданные участвуют в политике).
- Validator **MUST** проверить, что значение `Status` принадлежит множеству, заданному для этого `doc_type` в разделе «Type-aware status model» / «Allowed transitions».
- Validator **MUST** выдавать нарушения политики в терминах нормализованной severity (см. ниже), без подмены gate-решением.
- Validator **MUST NOT** объявлять статус недопустимым на основании правил, не закреплённых в Policy Contract для соответствующего `doc_type`.

## Gate contract

- Gate **MUST** принимать на вход результат validator, уже содержащий (или однозначно сводимый к) нормализованные уровни severity.
- Gate **MUST** применять политику блокировки/допуска (например fail / warn / pass) **только** к нормализованным уровням и к явным правилам контракта; gate **MUST NOT** вводить собственную независимую карту «finding → severity».
- Gate **MUST NOT** переопределять допустимость статуса по `doc_type`; спор по статусу разрешается только в пользу определений Policy Contract.

## Severity normalization rules

- Множество уровней severity для потребителей (включая gate) **MUST** быть фиксированным конечным набором (например: `blocker`, `warning`, `info` — точные имена могут быть унифицированы в одном месте политики; смысл: исчерпывающая классификация без дублирования независимых шкал).
- Каждое правило validator **MUST** относить находку к ровно одному нормализованному уровню согласно Policy Contract.
- Gate **MUST NOT** повышать или понижать severity находки в обход таблицы нормализации; допускается только агрегация итога прогона (например «есть blocker») по уже нормализованным уровням.
- Изменение таблицы severity **MUST** трактоваться как изменение Policy Contract (create-only новая версия архитектурного артефакта).

## Source-of-truth rule

- Нормы статусов, переходов, severity и разделения ролей validator/gate для жизненного цикла **MUST** браться из канонического Policy Contract (настоящий документ и его преемники по `Parent` / новым `DOA-ARCH-*`).
- Производные артефакты (включая человекочитаемые отчёты Report Layer) **MUST NOT** быть источником правды для политики статусов; они **MAY** отображать итог применения политики, но **MUST NOT** расширять или ослаблять нормы контракта.

## Architectural invariants

- **Create-only** для канона документов сохраняется: изменение политики **MUST** выполняться новым архитектурным документом, а не правкой исторического текста в нарушение репозиторной политики (если иное не оговорено отдельным решением репозитория).
- Единый Policy Contract для validator и gate — **инвариант**; расхождение определений **MUST** считаться дефектом архитектуры или внедрения, а не допустимым состоянием.
- Глобальный универсальный enum статусов для всех `doc_type` — **запрещён** как нормативная модель (**MUST NOT**).

## Traceability

- Родительский артефакт: `DOA-ARCH-017` (Report Generator MVP — граница отчёта vs SoT).
- Базовая модель статусов по типам: `DOA-ARCH-015`; нормативные решения: `DOA-DEC-037`, `DOA-DEC-038` — связаны по смыслу; при конфликте уточнения **MUST** вводиться новым архитектурным документом с явным `Parent`.

## MVP constraints

- В MVP охват **MUST** включать как минимум `doc_type`, перечисленные в разделе «Allowed transitions by doc_type»; прочие типы остаются вне нормы до расширения контракта.
- Полный граф зависимостей между всеми документами репозитория для gate **MAY** быть упрощён; минимально **MUST** соблюдаться статусная политика и нормализация severity.
- Report Layer **MUST NOT** требоваться для выполнения validator или gate; отчёт **MAY** использовать их вывод как входные данные, вне объёма настоящего контракта.
