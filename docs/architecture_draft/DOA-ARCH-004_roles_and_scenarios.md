## 2. Роли и сценарии (детализация)

## 2.1 Роли

### Primary User (основной пользователь)
Описание:
- управляет проектом через документацию
- не работает напрямую с кодом без документации
- принимает решения и фиксирует их

Ответственности:
- создание документов
- ведение planning layer
- фиксация решений (decision_log)
- контроль связности системы

Ограничения:
- не может выполнять задачи вне operational_plan
- не может изменять существующие документы (create-only)

---

### System (DocOps правила)
Описание:
- не отдельный пользователь, а набор ограничений

Функции:
- enforce docs-first
- enforce planning-first
- enforce create-only
- enforce traceability

---

## 2.2 Ключевые сценарии

### Сценарий 1: Создание проекта

Шаги:
1. Создаётся idea_to_mvp
2. Определяется scope проекта
3. Создаётся первый midterm_plan

Результат:
- проект формализован
- есть начальная структура

---

### Сценарий 2: Формирование ТЗ

Шаги:
1. Создаётся architecture_draft
2. Фиксируется структура системы
3. Определяются правила и ограничения

Результат:
- есть полное ТЗ
- можно переходить к детализации

---

### Сценарий 3: Планирование работы

Шаги:
1. Создаётся/обновляется operational_plan (create-only через новую версию)
2. Добавляются задачи с ID
3. Устанавливаются статусы

Результат:
- вся работа описана через задачи
- нет "неформальной" работы

---

### Сценарий 4: Принятие решения

Шаги:
1. Возникает новая идея/изменение
2. Создаётся decision_log
3. Обновляется planning при необходимости

Результат:
- решение зафиксировано
- есть контекст и обоснование

---

### Сценарий 5: Выполнение задачи

Шаги:
1. Выбирается задача из operational_plan
2. Выполняется работа
3. Создаётся implementation_snapshot

Результат:
- есть зафиксированный результат
- обеспечена traceability

---

### Сценарий 6: Проверка системы

Шаги:
1. Проверяются связи документов
2. Проверяется соответствие правилам
3. Создаётся audit_check

Результат:
- система остаётся согласованной
- нет "висячих" элементов

---

## 2.3 Ограничения сценариев

Запрещено:
- выполнять действия без operational_plan
- принимать решения без decision_log
- изменять существующие документы
- создавать документы вне структуры doc_types

Обязательные требования:
- каждая задача имеет ID
- каждый документ имеет metadata
- каждый результат зафиксирован snapshot

---

## 2.4 Связь сценариев

Flow:

idea_to_mvp
→ architecture_draft
→ midterm_plan
→ operational_plan
→ execution
→ implementation_snapshot
→ audit_check

decision_log может появляться на любом этапе
