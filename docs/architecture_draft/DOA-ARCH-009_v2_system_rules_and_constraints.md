## 9. System Rules and MVP Constraints

## 9.0 Summary
Раздел фиксирует базовые правила системы DocOps Agent и ограничения MVP.

---

## 9.1 Цель раздела
- зафиксировать правила работы системы
- определить применение в MVP
- ограничить scope системы

---

## 9.2 System Rules

### Rule 1 — Docs-first

**Определение**
Любое действие должно быть отражено в документах.

**Практика**
- нет “работы в голове”
- результат фиксируется через doc_types

**Check**
- есть ли документ, отражающий текущее состояние?

---

### Rule 2 — Planning-first

**Определение**
Работа выполняется только через план.

**Практика**
- задачи берутся из `operational_plan`
- нельзя выполнять задачи вне плана

**Исключение**
- действует только до создания первого accepted `operational_plan`

**Check**
- существует ли активный operational_plan?

---

### Rule 3 — Create-only

**Определение**
Содержательные изменения выполняются только через новый документ.

**Практика**
- update запрещён для смысловых изменений
- новая версия → новый ID
- история сохраняется

**Допуск**
- технические правки (опечатки, формат) допустимы в draft

**Check**
- не изменяется ли смысл существующего документа?

---

### Rule 4 — Explicit decisions

**Определение**
Все значимые решения фиксируются явно.

**Практика**
- используется `decision_log`

**Check**
- есть ли decision_log для изменения?

---

### Rule 5 — Traceability

**Определение**
Любое действие должно быть отслеживаемым.

**Практика**
- используются id, parent, replaces

**Check**
- можно ли восстановить цепочку?

---

## 9.3 Enforcement в MVP

- правила НЕ автоматизированы
- соблюдаются пользователем вручную
- контроль через validation + audit_check

---

### 9.3.1 Ответственность пользователя
Пользователь:
- создаёт документы
- выполняет validation
- выполняет audit_check
- несёт ответственность за корректность

---

## 9.4 MVP Constraints

### 9.4.1 Нет автоматизации
- нет авто-валидации
- нет авто-связывания

### 9.4.2 Нет execution engine
- нет workflow
- выполнение вручную

### 9.4.3 Нет UI
- markdown-only

### 9.4.4 Нет интеграций
- нет API
- нет sync

### 9.4.5 Ограничение масштаба
- один пользователь
- один проект

---

## 9.5 Архитектурные ограничения

- нет version control логики
- нет conflict detection

**Mitigation**
- audit_check
- decision_log

---

## 9.6 Нарушения

- задача вне operational_plan (кроме этапа идеи/ТЗ)
- изменение документа
- отсутствие metadata
- нарушение naming

---

## 9.7 Исправление

- создать decision_log
- создать корректный документ
- при необходимости пометить старый как obsolete
- выполнить audit_check

---

## 9.8 Definition of Done

- правила описаны
- есть check
- есть ограничения
- описаны нарушения и исправления
