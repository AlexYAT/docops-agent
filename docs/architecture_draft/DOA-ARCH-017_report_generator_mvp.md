## Report Generator MVP (DOA-ARCH-017)

## Summary

Определить MVP генератора отчетов для DocOps Agent.

Генератор должен строить человекочитаемый отчет на основе документов системы. Цель отчета — объяснить текущее состояние системы, показать ключевые связи между документами и дать понятный итог без необходимости читать все документы вручную.

## Context

В системе уже определен Report Layer как производный слой, а также зафиксировано новое Definition of Done:

- система проходит validator / gate
- на основе документов можно сгенерировать понятный человеку отчет

Для перехода от архитектурной идеи к реализации требуется определить минимальную модель генератора отчета.

## Goal

Создать MVP-модель генератора отчетов, которая:

- работает поверх существующих документов
- не изменяет Source of Truth
- выдает понятный человеку markdown-отчет
- ориентируется не на список файлов, а на объяснение состояния системы

## Non-Goals

В MVP не входят:

- Web UI
- интерактивная навигация
- редактирование документов
- автоматическое исправление ошибок
- сложная аналитика
- сохранение отчета как нового документа системы

## Input

Генератор отчета использует документы репозитория как входные данные.

Минимально учитываются:

- planning документы
- decision_log
- architecture_draft
- implementation_snapshot
- audit_check
- результаты validator / gate при наличии

## Output

Выход генератора — markdown-отчет, который:

- не является Source of Truth
- не фиксируется как create-only документ
- может пересобираться при изменении системы
- должен быть понятен человеку без знания внутренней структуры DocOps Agent

## Report Model

MVP-отчет должен строиться вокруг логики, понятной человеку, а не вокруг списка файлов.

Минимальные разделы отчета:

### 1. Общее состояние

Краткое объяснение:

- что это за система
- на каком этапе она находится
- каков текущий статус

### 2. Что запланировано

Сводка по planning-документам:

- ключевые направления
- активные или planned элементы
- ближайшие шаги

### 3. Какие решения приняты

Сводка по decision_log:

- только значимые решения
- формулировка человеческим языком
- влияние решений на систему

### 4. Что уже реализовано

Сводка по implementation_snapshot / audit_check / другим фактам реализации:

- что реально сделано
- что подтверждено
- что изменило состояние системы

### 5. Связи между слоями

Ключевой раздел MVP.

Отчет должен уметь показывать цепочки вида:

- идея → план → решение
- архитектура → решение → реализация
- план → реализация → текущее состояние

Не требуется полный graph view. Достаточно показать основные смысловые связи.

### 6. Текущее состояние системы

Сводка:

- что уже стабильно
- что находится в переходном состоянии
- какие ограничения остаются

### 7. Вывод для человека

Краткий финальный блок:

- что сделано на текущий момент
- что это дает
- что логично делать дальше

## Human-Readable Principle

Отчет должен отвечать не на вопрос: «какие документы существуют?»

А на вопрос: «что происходит в системе и что уже достигнуто?»

Это обязательное требование MVP.

## Minimal Generation Logic

MVP генератор может работать по упрощенной логике:

1. собрать документы известных типов
2. извлечь базовые metadata
3. определить связи Parent / ID / тип документа
4. сгруппировать документы по смысловым ролям
5. сформировать итоговый markdown-отчет по шаблону

Допускается, что первая версия будет использовать простые эвристики вместо полной semantic-модели.

## Quality Criteria

MVP считается архитектурно достаточным, если генератор способен:

- собрать отчет без ручной доработки
- показать хотя бы основные связи между документами
- выдать текст, понятный человеку
- не подменять Source of Truth
- не нарушать create-only модель

## Risks

- слабая связность старых документов
- неполные metadata
- сложность правильной группировки документов
- риск скатиться в «список файлов» вместо объясняющего отчета

## Notes

- MVP должен быть простым
- приоритет: понятность отчета человеку
- дальнейшее развитие возможно в сторону CLI, затем Web UI
- генератор отчета является частью Report Layer, но не частью Source of Truth
