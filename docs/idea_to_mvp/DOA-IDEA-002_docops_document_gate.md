## Summary

**DocOps Document Gate** — идея единого **механического контрольного шлюза** для репозитория DocOps Agent: перед фиксацией изменений (например commit/merge) система прогоняет проверки целостности документов и ссылок, выдаёт **machine-readable** отчёт и итоговый статус **ACCEPT** / **ACCEPT_WITH_WARNINGS** / **REJECT**, без семантических правок содержимого и с ограничением автоматических исправлений только **механическими** действиями.

## Problem

- Ошибки **ID**, **Parent**, путей и дубликатов обнаруживаются поздно или вручную.
- Неканонические «снимки идей» с **коллидирующими ID** (например повторное использование **DOA-IMP-013**) ломают трассируемость, если их считать источником истины.
- Нужен **предсказуемый** слой качества, согласованный с **create-only**: gate **не переписывает смысл** документов.

## Proposed solution

Ввести **Document Gate** как отчётливый компонент (CLI или хук), который:

1. Сканирует согласованный охват (как минимум `docs/**/*.md` и согласованные корневые файлы).
2. Выполняет проверки (включая уникальность **ID** по репозиторию, разрешимость ссылок по контракту, разделение canonical vs legacy по политике).
3. В **dry-run** не изменяет файлы и не блокирует среду — только отчёт и код выхода (или тот же отчёт без побочных эффектов).
4. Пишет **JSON-отчёт** стабильной схемы для CI и человека.
5. Возвращает один из статусов: **ACCEPT**, **ACCEPT_WITH_WARNINGS**, **REJECT** (правила отображения severity → статус задаются политикой, не «на глаз»).
6. **Автофиксы** допускаются **только механические** (например нормализация пробелов в metadata, единообразие перевода строк — перечень белого списка); **семантических** правок текста, заголовков и решений **нет**.

## MVP scope

### In

- **Dry-run mode:** полный прогон проверок без записи в файлы (кроме явно разрешённого отдельным флагом режима autofix, если он будет добавлен позже).
- **Machine-readable JSON report** с версией schema, временем прогона, списком находок и агрегатами.
- **Unique ID check across repository:** один логический **DOA-…** не может принадлежать двум каноническим источникам без явной ошибки/предупреждения по политике.
- **Статусы итога:** **ACCEPT** (нет блокирующих нарушений), **ACCEPT_WITH_WARNINGS** (есть предупреждения, блокирующих нет), **REJECT** (есть блокирующие нарушения по политике).
- **No semantic edits** со стороны gate.
- **Mechanical autofixes only** (белый список; по умолчанию выключены в MVP).

### Out

- Полная семантическая нормализация документов или «AI-редактура».
- Обязательная интеграция в конкретный CI-провайдер в MVP идеи (достаточно контракта CLI/exit code + JSON).
- Замена **decision_log** или **audit_check** как процессов.

## Constraints

- **Create-only** для содержательной истории: gate **не** заменяет политику новых файлов вместо правок.
- Соблюдение **DOA-DEC-031:** metadata — SoT для канонических документов; legacy учитывается отдельно.
- Любые автоматические изменения — только из явного **механического** набора и с отдельным режимом (не по умолчанию).

## Success criteria

- Политика **ACCEPT / ACCEPT_WITH_WARNINGS / REJECT** документирована и воспроизводима на одном и том же дереве файлов.
- JSON-отчёт парсится без догадок (версия schema в отчёте).
- Проверка **уникальности ID** находит коллизии между каноническими объявлениями.
- Dry-run не оставляет неожиданных следов в рабочей копии.

## Risks

- Жёсткий gate без учёта legacy «заклинит» репозиторий — нужна согласованность с **DOA-OP-019** и политикой warnings.
- Расширение белого списка **autofix** может скользнуть к семантическим изменениям — граница должна контролироваться явно (decision/architecture при расширении).

## Relation to current validator track

- **DOA-DEC-031** задаёт канонический контракт ссылок и метаданных и разделение canonical / legacy; Document Gate должен **опираться** на этот контракт при классификации нарушений и статусов.
- **DOA-OP-018** ввёл первую реализацию **link/ID validator** (**v0.1.0**) и отчёт первого прогона; Gate логично **наследует** или **оборачивает** эту логику как «продакшен-оболочку» с явными итоговыми статусами.
- **DOA-OP-019** расширяет правила (**v0.2**): folder/family для legacy, реестр ID, ссылки в `docs/`, severity; Gate должен **потреблять** те же категории/severity, когда **v0.2** будет готов, чтобы не расходиться с отчётами.
- **DOA-IMP-013** фиксирует **фактический** первый прогон и метрики **v0.1**; любые старые материалы с коллизией **DOA-IMP-013** **не** являются SoT — настоящий снимок с этим ID остаётся **`DOA-IMP-013_link_id_validator_first_run.md`**. Идея Gate оформлена **заново** в **DOA-IDEA-002** с **уникальным IDEA-ID**.
