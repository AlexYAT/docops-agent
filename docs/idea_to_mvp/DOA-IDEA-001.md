## Summary

DocOps Agent как **docs-first** система управления разработкой: единый слой документов, планирования и явных правил вместо неформального контекста.

## Goal

Убрать хаос и зависимость от «контекста в голове»: решения, планы и состояние системы должны быть **явными, прослеживаемыми и воспроизводимыми** для одного владельца/команды на этапе MVP.

## MVP Scope

### In

- Документы в Markdown с **metadata**, **naming** (`DOA-<TYPE>-<NNN>`) и договорённой структурой по **doc_types**
- **Planning layer**: longterm / midterm / operational планы + связь Parent
- **Правила**: create-only, planning-first, traceability (см. `WORKFLOW.md`)
- Каталоги `docs/` и `templates/` для типов артефактов

### Out

- Автоматизированный CI/CD пайплайл документов
- Мульти-пользовательский workflow с ролями и ACL
- Интеграция с трекером задач как обязательный источник правды

## Core Idea

Один человек или малая команда ведёт продукт через **явные артефакты**: от идеи и архитектурного черновика до операционного плана и снимков внедрения; значимые решения фиксируются в **decision_log**, проверки — в **audit_check**.

## Constraints

- **Markdown-only** для человекочитаемости и версионирования в Git
- Один основной пользователь сценария на этапе bootstrap
- Минимум внешних инструментов; Notion/MCP — опционально для публикации

## Success Criteria

- Можно создать полный **сквозной пример** (idea → … → audit) без противоречий в metadata и Parent
- Шаблоны в `templates/` используются для создания новых документов без «догадок по полям»
- **Legacy**-артефакты не переписываются; новые нарушения не маскируются
