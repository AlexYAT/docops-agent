# WORKFLOW

## Overview

- **DocOps Agent** — docs-first система: договорённости, планы и факты фиксируются в Markdown в каталоге `docs/`.
- **Содержательная работа** ведётся через документы согласованных **doc_types**; контекст не должен оставаться только неформальным.
- **Planning layer обязателен**: уровни LT / MT / OP задают рамку исполнения; исключения — в разделе [Planning rules](#planning-rules).
- **Create-only**: смена смысла — через **новый файл и новый ID**, без переписывания утверждённой истории.

## Document flow

**Основной поток** (каждый пункт — следующий этап по смыслу цепочки):

1. **idea_to_mvp**
2. **architecture_draft**
3. **longterm_plan** _(опционально)_
4. **midterm_plan**
5. **operational_plan**
6. **implementation_snapshot**
7. **audit_check**

Эквивалентно в виде цепочки:  
idea_to_mvp → architecture_draft → longterm_plan (optional) → midterm_plan → operational_plan → implementation_snapshot → audit_check

**Отдельно:** **`decision_log`** используется **на любом этапе** для явной фиксации значимых решений (см. [Decision rules](#decision-rules)).

## doc_types

### idea_to_mvp

- **Зачем:** зафиксировать идею, цель MVP и границы In/Out до глубокой детализации.
- **Когда:** старт линии или смена направления MVP.
- **Что фиксирует:** цель, scope, ограничения, критерии успеха, ближайшие шаги.

### architecture_draft

- **Зачем:** чертёж системы документов (doc_types, metadata, naming, правила).
- **Когда:** после IDEA или при смене базовых договорённостей.
- **Что фиксирует:** scope, роли/сценарии, модели документов и метаданных, workflow, DoD для «системы документов».

### longterm_plan

- **Зачем:** стратегический горизонт и крупные вехи.
- **Когда:** опционально, когда горизонт шире среднесрочного плана.
- **Что фиксирует:** стратегическую цель, горизонт времени, milestones, риски, ожидаемый результат.

### midterm_plan

- **Зачем:** разбить стратегию на вехи с ID, названием и критерием «done when».
- **Когда:** после LT (если есть) или от согласованной точки входа — до операционного цикла.
- **Что фиксирует:** цель на средний срок, вехи, зависимости, риски, ожидаемый результат.

### operational_plan

- **Зачем:** текущий цикл работ — задачи с ID, описанием, приёмкой и зависимостями.
- **Когда:** исполнение; основной разрешённый слой после принятия правил планирования.
- **Что фиксирует:** goal, scope, tasks, DoD цикла, заметки.

### implementation_snapshot

- **Зачем:** снимок «как сейчас»: сделано, связи, проблемы, следующий шаг.
- **Когда:** после значимого блока работ или по завершении OP-цикла.
- **Что фиксирует:** текущее состояние, выполненное, связь с задачами/коммитами (если есть), issues, next step.

### audit_check

- **Зачем:** ручная проверка metadata, naming, цепочек Parent; фиксация нарушений и legacy.
- **Когда:** после набора новых документов или перед закрытием этапа.
- **Что фиксирует:** охват, проверенные документы, findings, violations, рекомендации, нужные решения, итоговый result.

### decision_log

- **Зачем:** явный выбор из альтернатив, обоснование и trade-offs.
- **Когда:** при любом **значимом** решении по методологии, стеку, структуре и т.п.
- **Что фиксирует:** контекст, варианты, выбранное решение, почему, последствия, follow-ups.

## Planning rules

- Работа **вне `operational_plan` запрещена**, кроме **bootstrap**: **idea_to_mvp** → **architecture_draft** и подготовка планов **до первого accepted `operational_plan`**.
- После принятия первого **accepted** `operational_plan` **основная** исполнительная работа ведётся в рамках актуальных задач **OP**.
- Планирование по уровням: **`longterm_plan` → `midterm_plan` → `operational_plan`** (LT опциональна; без LT у MT — согласованный parent, например IDEA/ARCH).

## Decision rules

- **Значимые решения** фиксируются в **`decision_log`** (новый документ `DOA-DEC-<NNN>`).
- Решение **имеет parent-контекст** (архитектура, план или другой родительский артефакт).
- Без записи в **`decision_log`** содержательное изменение базовых договорённостей **невалидно** в рамках методологии (опора только на «устное» — недопустима).

## Create-only principle

- Утверждённые / используемые как источник правды документы **не переписываются содержательно**.
- Смена смысла → **новый документ**, **новый ID** (новый DEC / IMP / ARCH и т.д.).
- **Технические** правки (опечатки, формат) **до** первого использования как SoT — по договорённости; при сомнении — новый ID.

## Traceability

- У документа в metadata задаётся **`Parent`** (у корня линии — `null`).
- Цепочка уровней: **idea → architecture → (longterm?) → midterm → operational → snapshot → audit**; **decision_log** подключается в нужной точке с parent на контекст.
- Состояние и причины решений **восстанавливаются по документам** и связям Parent.

## Validation (manual, MVP)

Вручную (без обязательного CI):

- проверка **metadata** (Project, Doc type, ID, Status, Date, Parent);
- проверка **naming** (`DOA-<TYPE>-<NNN>` в соответствии с типом);
- проверка **связей** (логичность Parent и типов);
- итог формальной проверки — оформление **`audit_check`**.

## Typical workflow example

Шесть шагов сквозного сценария:

1. Создать **idea_to_mvp** (IDEA).
2. Создать **architecture_draft** (ARCH), parent — IDEA.
3. При необходимости **longterm_plan**, затем **midterm_plan** и **operational_plan** с задачами.
4. Выполнить задачи OP; значимые решения — отдельные **decision_log** с parent на контекст.
5. Зафиксировать **implementation_snapshot**.
6. Провести **audit_check** и зафиксировать result (включая legacy без «тихих» правок архива).
